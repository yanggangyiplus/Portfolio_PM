# Social Login Community Service - 의사결정 과정 문서

## 의사결정 개요

프로젝트 진행 중 주요 의사결정 사항과 그 근거를 문서화한 자료입니다.

## 주요 의사결정 사항

### 1. 기술 스택 선택

#### 의사결정: React + Node.js + PostgreSQL

**배경**:
프로젝트 초기 단계에서 프론트엔드와 백엔드 기술 스택을 결정해야 했습니다.

**대안 검토**:

| 대안 | 장점 | 단점 | 점수 |
|------|------|------|------|
| **React + Node.js** | • JavaScript 통일<br>• 빠른 개발 속도<br>• 풍부한 생태계 | • 서버 부하 가능성 | **9/10** |
| Vue.js + Python Django | • Python 생태계<br>• Django ORM | • 학습 곡선<br>• 기술 스택 분리 | 7/10 |
| Angular + Java Spring | • 엔터프라이즈급<br>• 안정성 | • 과도한 복잡도<br>• 개발 속도 느림 | 6/10 |

**결정 근거**:
1. 팀 내 React 경험자 다수로 학습 비용 최소화
2. JavaScript 통일로 개발 효율성 향상
3. 빠른 MVP 개발이 목표였으므로 개발 속도 우선

**트레이드오프**:
- 장점: 빠른 개발, 낮은 학습 비용
- 단점: 서버 부하 시 스케일링 필요 (향후 마이크로서비스 전환 고려)

**결과**:
- 개발 기간 20% 단축
- 팀 생산성 향상

### 2. 소셜 로그인 제공자 선택

#### 의사결정: 카카오, 네이버, 구글 3개 제공자 지원

**배경**:
한국 시장에서 어떤 소셜 로그인 제공자를 지원할지 결정해야 했습니다.

**시장 조사 결과**:
- 카카오 사용률: 65% (한국 1위)
- 네이버 사용률: 25%
- 구글 사용률: 10%

**대안 검토**:

| 대안 | 예상 사용률 | 개발 난이도 | 비용 |
|------|-----------|------------|------|
| 카카오만 지원 | 65% | 낮음 | 무료 |
| 카카오 + 네이버 | 90% | 중간 | 무료 |
| **카카오 + 네이버 + 구글** | **100%** | **높음** | **무료** |

**결정 근거**:
1. 사용자 편의성 극대화 (모든 사용자 지원)
2. 개발 비용은 증가하지만 사용자 획득 비용(CAC) 감소
3. 경쟁사 대비 차별화 포인트

**트레이드오프**:
- 장점: 모든 사용자 지원, 높은 전환율
- 단점: 개발 시간 30% 증가, 유지보수 복잡도 증가

**결과**:
- 소셜 로그인 전환율 78.5% 달성 (업계 평균 60% 대비 우수)
- 사용자 만족도 4.5/5.0

### 3. 데이터베이스 선택

#### 의사결정: PostgreSQL 선택

**배경**:
관계형 데이터베이스와 NoSQL 중 선택해야 했습니다.

**요구사항 분석**:
- 사용자-게시글-댓글 관계 모델링 필요
- 트랜잭션 보장 필요 (좋아요, 댓글 작성 등)
- 복잡한 쿼리 필요 (검색, 필터링)

**대안 검토**:

| 대안 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| **PostgreSQL** | • 관계형 데이터 지원<br>• 트랜잭션 보장<br>• 복잡한 쿼리 지원 | • 수평 확장 제한 | **9/10** |
| MongoDB | • 수평 확장 용이<br>• 유연한 스키마 | • 관계 모델링 어려움<br>• 트랜잭션 제한 | 5/10 |
| MySQL | • 안정성<br>• 성능 | • 기능 제한 | 7/10 |

**결정 근거**:
1. 관계형 데이터 구조가 명확함
2. 트랜잭션 보장이 필수적
3. 향후 검색 기능 확장 시 PostgreSQL의 Full-text Search 활용 가능

**트레이드오프**:
- 장점: 데이터 일관성, 복잡한 쿼리 지원
- 단점: 수평 확장 시 샤딩 필요 (향후 고려)

**결과**:
- 데이터 일관성 100% 유지
- 복잡한 검색 쿼리 성능 우수

### 4. 인프라 선택

#### 의사결정: AWS 선택

**배경**:
클라우드 인프라 제공자를 선택해야 했습니다.

**대안 검토**:

| 대안 | 장점 | 단점 | 비용 (월) |
|------|------|------|----------|
| **AWS** | • 서비스 다양성<br>• 한국 리전<br>• 문서화 우수 | • 복잡도 높음 | **200만원** |
| Google Cloud | • 머신러닝 강점<br>• 가격 경쟁력 | • 한국 리전 제한 | 180만원 |
| Azure | • 엔터프라이즈 통합<br>• Microsoft 생태계 | • 한국 시장 점유율 낮음 | 220만원 |

**결정 근거**:
1. 한국 리전(서울)으로 낮은 레이턴시
2. 풍부한 서비스 (EC2, RDS, S3, CloudFront)
3. 팀 내 AWS 경험자 존재

**트레이드오프**:
- 장점: 안정성, 확장성, 다양한 서비스
- 단점: 초기 설정 복잡도, 비용 관리 필요

**결과**:
- 서버 가동률 99.8% 달성
- 평균 응답 시간 245ms 달성

### 5. 검색 기능 구현 방식

#### 의사결정: Elasticsearch 도입

**배경**:
게시글 검색 기능을 구현하기 위해 검색 엔진을 선택해야 했습니다.

**대안 검토**:

| 대안 | 검색 품질 | 구현 난이도 | 비용 |
|------|----------|------------|------|
| PostgreSQL Full-text Search | 낮음 | 낮음 | 무료 |
| **Elasticsearch** | **높음** | **중간** | **50만원/월** |
| Algolia | 높음 | 낮음 | 100만원/월 |

**결정 근거**:
1. 검색 품질이 사용자 경험에 핵심적
2. 향후 고급 검색 기능 확장 가능
3. Algolia 대비 비용 효율적

**트레이드오프**:
- 장점: 높은 검색 품질, 확장 가능성
- 단점: 추가 인프라 비용, 운영 복잡도 증가

**결과**:
- 검색 정확도 78% 달성
- 검색 API 응답 시간 456ms (목표 500ms 이내)

### 6. 모바일 최적화 전략

#### 의사결정: 반응형 웹 우선, 앱 개발은 후순위

**배경**:
모바일 사용자가 증가하면서 모바일 전략을 수립해야 했습니다.

**데이터 분석 결과**:
- 모바일 사용자 비율: 65%
- 모바일 이탈률: 35% (데스크톱 15% 대비 높음)

**대안 검토**:

| 대안 | 개발 시간 | 사용자 경험 | 비용 |
|------|----------|------------|------|
| 반응형 웹만 | 2주 | 중간 | 낮음 |
| **반응형 웹 + 앱 (후순위)** | **2주 + 8주** | **높음** | **중간** |
| 네이티브 앱 우선 | 12주 | 높음 | 높음 |

**결정 근거**:
1. 빠른 MVP 출시가 우선
2. 반응형 웹으로 모바일 커버 가능
3. 사용자 피드백 수집 후 앱 개발 결정

**트레이드오프**:
- 장점: 빠른 출시, 낮은 초기 비용
- 단점: 네이티브 앱 대비 사용자 경험 제한

**결과**:
- 반응형 웹으로 모바일 사용자 65% 커버
- 모바일 이탈률 35% → 18%로 개선 (반응형 웹 개선 후)
