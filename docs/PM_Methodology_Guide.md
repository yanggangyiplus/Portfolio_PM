# PM Methodology Guide

Product Manager / Planner / UX Designer 포트폴리오 작성 가이드

## 포트폴리오 구조 원칙

### 1. 기획 문서 중심
- 개발 코드는 배제하고 기획 문서, UX 설계, IA 중심으로 구성
- 실제 산출물과 문서를 중심으로 포트폴리오 구성

### 2. 프로젝트별 통일된 구조
각 프로젝트는 다음 6가지 구성으로 통일:
- **IA (서비스 구조도)**: 정보 구조도 및 ERD
- **UX_Flow**: 사용자 플로우 다이어그램
- **Wireframes (또는 Screens)**: 핵심 화면 와이어프레임
- **Requirements (REQ/FUNC)**: 기능명세서
- **Research (시장/사용자)**: 시장 분석 또는 사용자 연구
- **Presentation (PPT/PDF)**: 발표 자료

### 3. 폴더 기반 문서 관리
- Figma나 Notion 결과물도 폴더 기반으로 보관
- 각 산출물을 명확한 폴더 구조로 정리

## 프로젝트별 구조 가이드

### 커뮤니티/웹 서비스 프로젝트
```
프로젝트명/
├── IA/                     # 정보 구조도, ERD
├── UX_Flow/                # 사용자 플로우
├── Wireframes/              # 와이어프레임
├── Requirements/           # 기능명세서
├── Survey_Research/        # 사용자 조사
├── Roadmap/                # 로드맵
└── Presentation/           # 발표 자료
```

### 브랜드/디자인 프로젝트
```
프로젝트명/
├── IA/                     # 정보 구조도
├── UX_Flow/                # 사용자 플로우
├── Style_Guide/            # 디자인 시스템
├── Screens/                # 주요 화면 디자인
├── Editing_Mode_Design/    # 관리자 모드 설계
└── Presentation/           # 발표 자료
```

### UX 개선 프로젝트
```
프로젝트명/
├── User_Interviews/        # 사용자 인터뷰
├── PainPoint_Report/       # Pain Point 분석
├── UX_Flow/                # 개선된 플로우
├── Before_After/           # Before → After
├── Prototype/              # 프로토타입
└── Presentation/           # 발표 자료
```

### 게임 기획 프로젝트
```
프로젝트명/

### 서비스 전략 프로젝트
```
프로젝트명/
├── Market_Analysis/        # 시장 분석

## 문서 작성 가이드

### README.md 작성
각 프로젝트의 README.md에는 다음 내용을 포함:
- 프로젝트 개요
- 주요 기획 내용
- 산출물 목록
- 프로젝트 기간
- 역할
- 주요 성과

### IA (Information Architecture)
- 서비스 전체 구조도
- 페이지 구조도
- ERD (Entity Relationship Diagram)
- 데이터 구조 설계

### UX_Flow
- 핵심 사용자 플로우 다이어그램
- 예외 처리 플로우
- 상태 전이 다이어그램

### Wireframes
- 핵심 화면 와이어프레임
- 상호작용 명세
- 반응형 레이아웃 고려

### Requirements
- 기능명세서 (FUNC-001, FUNC-002 등)
- 요구사항 명세서
- API 명세서 (필요시)

### Research
- 사용자 조사 결과
- 시장 분석 보고서
- 경쟁사 분석
- 인사이트 도출

### Presentation
- 기획 발표 자료
- PPT 또는 PDF 형식
- 핵심 내용 요약

## 포트폴리오 제출 시 주의사항

### 1. 실제 산출물 중심
- 실제로 작업한 문서와 산출물을 중심으로 구성
- 가짜나 과장된 내용은 포함하지 않음

### 2. 명확한 역할 설명
- 각 프로젝트에서 본인의 역할을 명확히 명시
- 팀 프로젝트인 경우 기여도 명시

### 3. 일관된 톤앤매너
- 모든 문서가 동일한 사람이 작성한 것처럼 일관된 톤 유지
- 전문적이고 논리적인 문체 사용

### 4. 시각적 자료 활용
- 다이어그램, 와이어프레임, 스크린샷 등 시각적 자료 활용
- 텍스트만으로 구성하지 않고 시각적 요소 포함

## 다음 단계

포트폴리오 구조가 완성되었으니, 다음 단계를 진행하세요:

1. **기존 프로젝트 문서 정리**: 기존에 작업한 문서들을 해당 폴더에 정리
2. **README 작성**: 각 프로젝트의 README.md에 상세 내용 작성
3. **산출물 추가**: 각 폴더에 실제 산출물 파일 추가
4. **포트폴리오 완성도 검토**: 모든 프로젝트의 구조와 내용 확인
